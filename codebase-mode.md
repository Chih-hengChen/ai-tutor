# 源码阅读模式工作流

本文件为源码阅读模式的详细工作流。当 SKILL.md 判定进入源码阅读模式后，读取本文件执行。

## 适用场景

- "帮我梳理当前项目的登录鉴权模块原理"
- "理解这个仓库的状态管理是怎么实现的"
- "看一下这个项目的路由架构"
- 任何"理解当前工作目录中已有代码"的目标

## Phase 1: 项目扫描与规划

1. **恢复检测：** 检查 `./ai-tutor/records/` 下未完成的源码阅读模式记录。

2. **项目扫描：** 自动执行以下操作（静默，不逐条展示给用户）：
   - 读取目录结构（必须使用 `tree -I "node_modules|.git|dist|build|target|vendor|.next|.nuxt|coverage|__pycache__|.venv|venv|env|Pods|.gradle|.idea|.vscode|out|bin|obj" -L 3`，排除所有依赖和构建目录，深度限制 3 层，防止输出打爆上下文窗口）
   - 读取项目配置文件（`package.json` / `Cargo.toml` / `go.mod` / `pom.xml` 等，识别技术栈）
   - 识别入口文件和主要目录结构
   - 快速 grep 关键词定位目标模块的大致位置

3. **生成代码地图：** 基于扫描结果，生成一份针对当前项目的"代码地图"：
   - 用 ASCII 或 Mermaid 画出项目的模块关系和依赖方向
   - 标注目标功能涉及的文件和函数
   - 标注阅读顺序（从入口到细节）

4. **生成教学大纲：** 按阅读顺序拆分为节点，每个节点对应一组相关文件或一个核心流程：
   ```
   模块: 登录鉴权流程
     1.1 入口：路由层如何拦截未登录请求 (src/routes/auth.ts)
     1.2 JWT 签发逻辑 (src/services/auth/jwt.ts)
     1.3 Token 刷新机制 (src/services/auth/refresh.ts)
     1.4 中间件鉴权链 (src/middleware/auth.ts)
     1.5 用户角色权限判断 (src/utils/permission.ts)
   ```

5. **展示确认：** 展示代码地图和大纲，让用户确认或调整范围。

6. **写入文件：** 按源码阅读模式模板写入记录。

### 源码阅读模式记录模板

```markdown
# [项目名] 源码阅读记录
模式: 源码阅读模式
技术栈: [识别出的技术栈]
项目路径: [当前工作目录]
开始时间: YYYY-MM-DD

## [模块名]
- [ ] x.x [阅读节点] [未开始] — 涉及文件: xxx.ts, yyy.ts
- [x] x.x [阅读节点] [已掌握] — 1次通过

## 学习日志
| 日期 | 节点 | 结果 | 备注 |
|------|------|------|------|
```

## Phase 2: 引导阅读与讲解

1. **状态检查：** 读取记录，找到第一个待处理节点，标记 `[阅读中]`。

2. **定位代码：** 告诉用户打开哪个文件、关注哪些行/函数：
   - "打开 `src/middleware/auth.ts`，重点看第 23-47 行的 `verifyToken` 函数"
   - 如果文件较长，只标注关键区间，不要让用户读整个文件

3. **结构讲解：** 用精炼语言讲解这段代码的设计意图和核心逻辑：
   - **不要逐行翻译代码**，而是讲"这段代码解决什么问题、用什么思路解决的"
   - 用流程图或时序图展示调用链（参考 `visual-aids.md`）
   - 标注关键设计决策："这里用了双层缓存而不是单层，是因为..."

4. **考核：** 出一道**基于实际代码**的题目，而非脱离代码的理论题：
   - 追踪题："如果 Token 过期，代码是在哪一行触发刷新的？走的是什么调用链？"
   - 推理题："如果把第 35 行的 `&&` 改成 `||`，会发生什么？"
   - 定位题："新增一种角色 `admin`，需要改哪些文件？为什么？"
   - 调试题："假设用户反馈'刷新 Token 后还是被踢出'，你觉得可能是哪里的问题？在代码中定位。"

## Phase 3: 验证

与知识模式相同的分级验证机制：
- 概念性错误 → 换角度讲，标注代码中的关键行
- 细节错误 → 提示"再看一下第 X 行的判断条件"
- 接近正确 → 部分提示

用户回答时鼓励引用代码位置（"在第 XX 行"），养成读代码的习惯。

## Phase 4: 模块总结

1. **代码地图更新：** 回到 Phase 1 的代码地图，高亮已理解的模块。
2. **调用链总结：** 用 Mermaid 时序图串起该模块的完整调用链。
3. **设计模式归纳：** 总结代码中用到的设计模式或架构思想。
4. **个性化笔记：** 记录用户在阅读过程中的盲区和理解偏差。
5. **写入归档文件。**

## 源码阅读的特殊约束

1. **不替用户读代码：** 告诉用户看哪个文件哪几行，让用户自己读，导师负责引导和解释。
2. **关注设计意图：** 讲"为什么这样写"而不是"这行代码做了什么"。
3. **关联全局：** 每个节点讲完，帮用户在代码地图上定位"你现在在这里"，建立整体认知。
4. **渐进深入：** 先讲主干流程（正常路径），再讲分支逻辑（异常处理、边界情况）。
