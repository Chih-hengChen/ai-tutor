# 项目模式工作流

本文件为项目模式的详细工作流。当 SKILL.md 判定进入项目模式后，读取本文件执行。

## 适用场景

- "学 Node.js 做一个聊天室"
- "用 React 写一个 Todo App"
- "搭建一个个人博客"
- 任何以"做出一个可运行的东西"为目标的学习

## Phase 1: 需求分析与规划

1. **恢复检测：** 检查 `~/.claude/ai-tutor/records/` 下未完成的项目模式记录。
   - 有 → 展示进度，询问继续还是新项目。
   - 无 → 继续。

2. **需求澄清：** 和用户明确项目范围（2-3 轮对话即可，不要过度分析）：
   - 核心功能有哪些？
   - 技术栈是什么？（用户指定，或推荐 2-3 个方案供选择）
   - 用户对这些技术的熟悉程度？

3. **环境自动检查：** 根据技术栈，静默执行版本检查命令：
   - Node.js → `node -v`, `npm -v`
   - Python → `python --version`
   - Rust → `rustc --version`, `cargo --version`
   - Go → `go version`
   - Java → `java -version`
   - 检测结果处理：
     - 已安装 → 确认版本兼容，继续
     - 未安装 → 给出对应系统的安装命令（如 `winget install OpenJS.NodeJS` / `brew install node` / `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh`），等用户安装完成后继续
     - 版本过低 → 提示升级建议
   - 这一步在后台完成，只向用户展示结论（"检测到 Node.js v20.x ✓" 或 "未检测到 Rust，需要先安装"）

4. **生成项目规划：** 按里程碑拆分，遵循以下原则：
   - **第一个里程碑必须是 MVP**（最小可运行版本）
   - 每个里程碑产出一个**可运行、可演示**的增量
   - 后续里程碑逐步增加功能
   - 每个节点粒度："一次教学交互能完成的功能单元"

   示例（Node.js 聊天室）：
   ```
   里程碑1: 项目初始化与 HTTP 服务（MVP 能跑起来）
     1.1 初始化项目，配置基本目录结构
     1.2 Express 搭建 HTTP 服务器
     1.3 提供静态 HTML 页面

   里程碑2: WebSocket 实时通信
     2.1 WebSocket 原理 + 安装 socket.io
     2.2 服务端处理连接/断开事件
     2.3 客户端建立连接

   里程碑3: 聊天核心功能
     3.1 消息广播
     3.2 用户昵称
     3.3 在线用户列表

   里程碑4: 体验优化
     4.1 消息样式和时间戳
     4.2 私聊功能
     4.3 消息持久化
   ```

5. **展示确认：** 允许用户调整功能范围和顺序。

6. **写入文件：** 按项目模式模板写入记录。

## Phase 2: 功能教学与引导实现

1. **状态检查：** 读取记录，找到第一个待处理节点，标记 `[实现中]`。

2. **知识铺垫（精简）：** 只讲"实现这个功能需要知道的"：
   - 用户已有基础 → 跳过铺垫，直接给任务
   - 涉及新概念 → 用 2-3 句话 + 1 个代码片段说明用法
   - 不做长篇理论讲解

3. **实现任务：** 给出明确要求但不给完整代码：
   - 功能要求和预期行为
   - 提示关键 API 或模块名
   - 指出需要新建/修改的文件
   - 必要时给出代码骨架或接口定义
   - **不给完整实现**

4. **等用户动手：** 让用户自己写代码。用户卡住时按优先级引导：
   - 让用户看报错信息，尝试自行理解
   - 引导定位方向（"看看第 X 行附近的逻辑"）
   - 给具体提示（"这个 API 的第二个参数应该传什么？"）
   - 最后才给关键代码片段

## Phase 3: 功能验证

1. **运行验证：** 指导用户启动和测试：
   - 如何启动（`node server.js`、浏览器打开 localhost:3000 等）
   - 验证标准要具体："打开两个标签页，A 发消息，B 能看到"
   - 用户代码有报错 → 引导自己读报错定位，不直接修

2. **代码审查（轻量）：**
   - 功能是否正确实现
   - 是否有明显 bug 或边界问题
   - 能跑通、功能正确即可，不要求完美

3. **验证结果：**

   **未通过：**
   - 描述问题现象（不说答案）
   - 引导定位（"看一下 console 输出"）
   - 记入踩坑记录表
   - 等用户修复后重新验证

   **通过：**
   - 简短肯定
   - 代码有改进空间 → 作为"进阶建议"附后，不强制改
   - 更新记录 `[已完成]`，追加学习日志
   - 进入下一个节点

## Phase 4: 里程碑总结与项目文档沉淀

1. **功能回顾：** 这个里程碑实现了什么，在整体架构中的位置。

2. **知识点梳理：** 归纳过程中涉及的技术知识，结合项目代码讲解（不空洞复述理论）。

3. **踩坑总结：** 整理踩坑记录。

4. **架构图：** 用 Mermaid 或 ASCII 画出当前项目的架构/数据流（参考 `visual-aids.md`）。

5. **写入文件：** `~/.claude/ai-tutor/summaries/[项目slug]_[里程碑slug].md`，包含：
   - 功能说明和架构图
   - 关键代码片段及解释
   - 踩坑和解决方案
   - 技术知识点清单

6. **项目完成时额外输出：**
   - 完整 README（功能、技术栈、运行方式、项目结构）
   - 整体技术回顾
   - 进阶方向建议

7. **节奏建议：** 里程碑完成后提示休息。
