# 深度讲解模式工作流

本文件为深度讲解模式的详细工作流。当 SKILL.md 判定进入深度讲解模式后，读取本文件执行。

## 适用场景

需要理解复杂系统内部机制的主题，特征：
- 涉及大型源码库（React 源码、Vue 源码、Webpack 源码等）
- 运行时/引擎级原理（V8 引擎、JVM、Node.js 事件循环等）
- 编译器/解释器原理（Babel、TypeScript 编译器、LLVM 等）
- 操作系统内核机制（进程调度、内存管理、文件系统等）
- 任何以"源码级理解"、"底层原理"、"深入实现机制"为目标的学习

用户明确表达"深入理解"、"源码级"、"底层原理"、"实现机制"时触发。SKILL.md 在模式识别阶段发现主题属于系统级底层内容时也应路由到此模式。

## 与知识模式的核心区别

知识模式适合"学会用"（React Hooks 怎么用），深度模式适合"理解为什么这样设计"（React Hooks 的 Fiber 链表是怎么工作的）。

| 维度 | 知识模式 | 深度讲解模式 |
|------|---------|-------------|
| 粒度 | 最小知识点 | 最小可闭环机制 |
| 单节深度 | 精炼，控制篇幅 | 讲透，允许长输出 |
| 考核时机 | 每个知识点后立刻考核 | 一个主题块讲完后统一考核 |
| 源码引用 | 可选 | 必须（文件名、函数名、行号） |
| 可视化 | 辅助说明 | 每节必须有结构图或流程图 |

## 大纲策略：主题块拆分

不按"知识点"拆分，按"完整机制"拆分为**主题块**。每个主题块围绕一个核心问题展开，包含 5-8 个连续子节。

### 拆分原则

每个主题块必须是一个**可独立理解的完整闭环**：
- 有明确的"它解决什么问题"
- 有核心数据结构和关键字段
- 有完整的运行流程
- 能和前后主题块建立关联

### 拆分示例（React Fiber 架构）

不要把 "Fiber 架构" 当成一个小节。应该这样拆：

```
主题块: Fiber 架构
  1.1 Fiber 为什么出现 — Stack Reconciler 的瓶颈在哪
  1.2 FiberNode 的核心字段 — tag、type、return、child、sibling 等
  1.3 双缓存树 — current 树与 workInProgress 树的关系
  1.4 render 阶段如何遍历 — beginWork 和 completeWork 的交替
  1.5 commit 阶段如何提交 — mutation、layout、passive 三个子阶段
  1.6 调度和优先级如何接入 — lane 模型与 Scheduler 的交互

主题块: Hooks 实现
  2.1 Hooks 链表结构 — memoizedState 如何串联
  2.2 useState 的 mount 和 update — mountState vs updateState
  2.3 useEffect 的注册和执行 — mountEffect vs updateEffect
  ...
```

### 拆分示例（V8 引擎）

```
主题块: JavaScript 执行流水线
  1.1 从源码到字节码 — Parser → BytecodeGenerator
  1.2 Ignition 解释器 — 字节码执行主循环
  1.3 TurboFan 编译管线 — 热点检测到机器码生成
  1.4 内联缓存 (IC) — 如何加速属性访问
  1.5 隐藏类 — 对象形状与内存布局优化
```

## Phase 1: 摸底与规划

1. **恢复检测：** 检查 `./ai-tutor/records/` 下未完成的深度讲解模式记录。

2. **前置知识检测：** 评估该主题需要的核心前置知识。
   - 识别 1-3 个核心前置（如 React 源码需要：JS 原型链、链表、树遍历、闭包）
   - 出 1 道快速诊断题
   - 前置不足 → 建议补基础，但允许跳过（后续涉及该知识时自动补充更详细的背景解释）

3. **确认学习深度：** 询问用户期望的深度：
   - "理解设计思想和核心机制"（不需要追踪每一行代码）
   - "能跟着源码读下去"（需要具体的文件、函数、行号指引）
   - 根据回答调整引用源码的详细程度

4. **生成架构总览图：** 用 Mermaid 或 ASCII 画出整个系统的架构鸟瞰图，标注各子系统/模块的关系和数据流向。

5. **生成主题块大纲：** 按上述拆分原则，生成主题块和子节的大纲。每个子节标注一个核心问题。

6. **展示确认：** 让用户选择先学哪个主题块，允许调整范围。

7. **写入文件：** 按深度讲解模式模板写入记录。

### 深度讲解模式记录模板

```markdown
---
mode: deep-dive
start_date: YYYY-MM-DD
target_depth: conceptual | source-code
nodes:
  "1.1":
    name: [子节名]
    module: [主题块名]
    status: pending
    attempts: 0
    last_tested: null
    mastery_level: 0
    core_question: [这节要回答的核心问题]
---

# [名称] 学习记录

## 学习日志
| 日期 | 节点 | 结果 | 备注 |
|------|------|------|------|
```

## Phase 2: 深度讲解

这是深度模式的核心，每节必须按固定结构展开，不允许压缩成简短概述。

### 单节讲解结构（5 段式，每节必须全部包含）

**第 1 段：问题背景（这个机制解决什么问题）**
- 旧方案/旧实现有什么缺陷
- 新方案要达到什么目标
- 为什么现有机制不够用

**第 2 段：核心数据结构（涉及哪些关键对象/字段）**
- 列出核心数据结构，用 ASCII 或表格展示字段布局
- 标注每个字段的作用，以及字段之间的关系（指针、引用、索引）
- 如果源码中有 TypeScript/Flow 类型定义，引用真实类型

**第 3 段：运行流程（代码如何推进）**
- 用流程图或时序图展示执行路径
- 标注源码入口函数和关键调用链（`functionA() → functionB() → functionC()`）
- 给出一段伪代码总结核心逻辑（不是翻译全部代码，而是提炼骨架）
- 标注分支条件和边界情况

**第 4 段：与旧方案对比（为什么这样设计）**
- 新旧方案的差异和设计动机
- 这个设计带来的好处和牺牲
- 与同一系统中其他机制的关系

**第 5 段：常见误解和边界（容易踩的坑）**
- 列出 1-3 个常见误解，说明为什么错
- 标注边界条件和特殊场景（错误处理、并发、性能边界等）

### 讲解时的硬性要求

1. **必须引用真实源码**：标注文件路径和函数名（如 `packages/react-reconciler/src/ReactFiberBeginWork.js` 的 `beginWork()`），让用户能打开代码对照阅读
2. **必须有可视化**：每节至少包含一个结构图（ASCII）或流程图（Mermaid），用于展示数据结构关系或调用链
3. **必须有伪代码**：用简化的伪代码总结核心流程骨架，不要让用户对着几千行源码迷失
4. **字段关系必须说清**：涉及链表、树、指针等数据结构时，用箭头图标注节点之间的连接关系

### 讲解节奏

- 同一主题块内的子节连续讲解，不要在每节后都停下来考核
- 子节之间自然过渡："上一节我们看了 FiberNode 的字段，这些字段构成了链表树。现在我们看 React 是如何遍历这棵树的——"
- 一个主题块全部讲完后，统一进入 Phase 3 考核

## Phase 3: 主题块考核

一个主题块（5-8 个子节）全部讲完后，进行统一考核。

### 考核形式

出 2-3 道题，覆盖该主题块的核心机制：

1. **源码追踪题**（必须）：给出一个具体场景，让用户追踪调用链
   - "当 React 执行到 `beginWork(current, workInProgress, renderLanes)` 时，如果 workInProgress.tag 是 FunctionComponent，接下来会走哪个函数？整个调用链是什么？"

2. **机制推理题**（必须）：改变某个条件，让用户推理结果
   - "如果把 Fiber 的 return 指针去掉，会导致什么问题？在哪个具体环节会出错？"

3. **设计理解题**（可选）：关于设计决策的理解
   - "React 为什么选择用链表而不是数组来组织 Fiber 节点？如果用数组会有什么问题？"

### 评分与后续

- 全部回答到位 → 主题块通过，更新所有子节状态为 mastered
- 部分答对 → 只标记答对的子节为 mastered，未通过的子节需要补充讲解后再单独出题
- 全部未通过 → 回到该主题块，用不同角度重新讲解核心机制，然后重新考核

## Phase 4: 主题块总结

一个主题块通过后，生成深度总结文档：

1. **机制总结图**：用一张完整的 Mermaid 或 ASCII 图把该主题块所有子节的机制串联起来，展示端到端的数据流和控制流
2. **源码地图**：列出该主题块涉及的所有源码文件和关键函数，作为用户的"索引"，方便后续自己翻阅
3. **设计模式归纳**：总结源码中体现的设计模式或架构思想
4. **跨主题块关联**：标注这个主题块和其他主题块的接口点（"这个模块的输出是下一个模块的输入"）
5. **写入文件：** `./ai-tutor/summaries/[主题slug]_[主题块slug].md`

完成一个主题块后，回到总览图标注已完成区域，让用户选择下一个主题块。

## 深度讲解模式的特殊约束

1. **以讲透为优先**：不要因为"输出克制"而压缩讲解。宁可一节讲 2000 字把机制说透，也不要 500 字带过然后做题。
2. **源码级精度**：引用的函数名、文件路径、字段名必须准确。如果不确定具体位置，如实告诉用户"这个函数在 xxx 模块中，具体行号可能因版本不同"，不要编造。
3. **机制闭环**：每个子节必须形成"问题→结构→流程→对比→边界"的完整闭环，不能只讲一半。
4. **伪代码不是偷懒**：伪代码是帮助用户建立框架认知的工具，核心分支和关键判断必须保留，不能简化成"// 核心逻辑"就完事。
5. **鼓励用户对照源码**：讲解时主动提示"你可以打开 xxx.js 跟着看"，培养用户读源码的能力。
